Overview - A Shared Platform for Development and Testing

This platform provides readily available IDP/SP services for development and testing.

Never upload production data.

SAML Entities are identified by their Entity ID - use unique IDs to prevent overwrites.

The SAML BOX platform consists of two core components working together:

For the Service Provider (SP), we use Spring Security SAML 6.4.3. This handles the complete SAML flow when acting as SP, including:

  • Generating and customizing authentication requests
  • Validating IDP responses and extracting claims
  • Managing user sessions and Single Logout (SLO)

For the Identity Provider (IDP), we implement Keycloak 26.1.2 as Spring doesn't natively support IDP functionality. Keycloak provides:

  • Centralized identity management across protocols (SAML/OAuth/OIDC)
  • SP client configuration through its administration API
  • Flexible authentication flows and user federation

Platform Architecture

Metadata Management

SAML integration requires mutual metadata exchange between SP and IDP:

Both SP and IDP metadata can be downloaded from SAML BOX. Corresponding partner metadata must be uploaded to establish trust relationships. Metadata is identified by Entity ID - ensure uniqueness when creating new entities.

SAML BOX as Service Provider

SAML BOX supports both SP-initiated and IDP-initiated SSO flows. The SP-initiated flow follows this sequence:

  1. User accesses SP-protected resource
  2. SP generates SAML AuthnRequest
  3. User is redirected to IDP for authentication
  4. IDP returns SAML Response
  5. SP validates response and establishes session

Configurable SP Parameters

Request Bindings: Choose between HTTP POST (recommended) or HTTP Redirect binding for sending AuthnRequests. Legacy SOAP and Artifact bindings are not supported.

NameID Formats: Specify how subject identifiers should be interpreted. Supported formats include:

urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress
urn:oasis:names:tc:SAML:2.0:nameid-format:transient

Custom formats can be entered for special requirements.

Request Signing: While optional, signing AuthnRequests is recommended for production-grade implementations. Uses SP's private key for signing.

Session Management

Successful authentication creates a Spring Security session containing:

  • NameID and session index
  • Authentication instant
  • Assertion attributes

Session details are visible in the SAML BOX UI. New authentications overwrite existing sessions.

Single Logout (SLO)

SP-initiated SLO propagates logout through these steps:

  1. SP sends LogoutRequest to IDP
  2. IDP invalidates global session
  3. IDP sends LogoutResponses to registered SPs
  4. SAML BOX processes logout and clears local session

IDP-initiated SLO is currently not supported due to Keycloak limitations.

SAML BOX as Identity Provider

Powered by Keycloak's SAML implementation, the IDP functionality supports:

  • SP registration as Keycloak clients
  • Per-SP configuration of authentication flows
  • Flexible assertion mapping and attribute release

IDP-initiated SSO

Enables direct SP access without initial AuthnRequest:

  1. User authenticates directly at IDP
  2. IDP generates unsolicited SAML Response
  3. User is redirected to SP with valid assertion

Requires pre-configured SP ACS URL in Keycloak client settings.

Response Configuration

Keycloak supports multiple response configurations:

  • Bindings: HTTP POST or Redirect for Assertion Consumer Service
  • Signing: Configurable at assertion or response level
  • Encryption: Optional using SP's public certificate
